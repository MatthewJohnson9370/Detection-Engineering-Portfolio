# Investigation Guide — T1123 Audio Capture

## Purpose

This guide is intended for alerts generated by either of the following detections:

* Microphone ConsentStore registry activity
* PowerShell recording-device enumeration and selection

Neither alert confirms that audio was recorded. The goal of the investigation is to determine whether the activity is expected for the host and user, or whether it occurred alongside other suspicious behavior.

---

## Initial Triage

Start by identifying which detection triggered.

| Detection                                 | Initial Focus                                                                                        |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Microphone ConsentStore Registry Activity | Identify the process that modified the registry value and determine whether it is approved software. |
| PowerShell Recording Device Selection     | Review the full script block, execution path, parent process, and user context.                      |

For either alert, I would first collect:

* Hostname
* Username
* Process name and full path
* Parent process
* Command line or PowerShell script block
* Event timestamp
* Registry path or script content that caused the alert

---

## Host and User Context

I would determine whether the activity makes sense for the affected endpoint.

Questions I would ask:

* Is this a conferencing, dictation, clinical, accessibility, or shared workstation?
* Does the user normally use approved microphone-based software?
* Is the process signed and installed from an expected location?
* Is the host part of a known group that uses dictation or recording tools?
* Has the same process generated similar activity on this host before?

Expected activity may be more common on systems using Teams, Zoom, Dragon Medical, accessibility tools, or approved recording software.

---

## Review Related Process Activity

For registry-based alerts, I would review the Sysmon Event ID 13 details and identify the process responsible for the registry modification.

For PowerShell-based alerts, I would review the complete Event ID 4104 script block and then pivot into related process creation events.

I would pay close attention to:

* PowerShell launched from `Temp`, `Downloads`, user profile directories, or network shares
* Encoded commands, hidden windows, or suspicious execution flags
* Unexpected parent processes such as Office applications, browsers, script hosts, or remote management tools
* Unsigned or unfamiliar executables
* Activity executed by a service account or privileged account without a known business reason

---

## Review Nearby Activity

I would review activity from approximately 15 minutes before and after the alert.

Useful telemetry includes:

| Telemetry                | Why It Matters                                        |
| ------------------------ | ----------------------------------------------------- |
| Sysmon Event ID 1        | Process creation and parent-child relationships       |
| Sysmon Event ID 3        | Outbound network connections from the related process |
| Sysmon Event ID 11       | Creation of possible audio files or archives          |
| PowerShell Event ID 4104 | Full script content and follow-on PowerShell activity |
| Sysmon Event ID 13       | Additional registry changes made by the same process  |

I would look for files such as `.wav`, `.mp3`, `.m4a`, `.aac`, or `.flac`, but the absence of those extensions does not rule out suspicious collection.

---

## Escalation Indicators

I would escalate the alert when microphone-related activity occurs with one or more of the following:

* PowerShell from an unusual script path or parent process
* Encoded or obfuscated PowerShell content
* Unrecognized or unsigned recording-related software
* Audio file or archive creation shortly after the event
* Suspicious outbound network traffic from the same process
* Activity on a host where microphone usage is not expected
* Similar activity affecting multiple endpoints in a short period

---

## Benign Disposition

I would close the alert as expected activity when the process, user, host role, and timing all align with a verified business function.

Examples may include approved conferencing software, clinical dictation tools, accessibility applications, or documented endpoint-management automation.

Before tuning, I would document the verified process path, publisher, host group, account, and reason the activity is expected.

---

## Tuning Feedback

I would not create broad exclusions based only on one benign alert.

If repeated verified activity creates noise, I would recommend narrow tuning based on specific evidence, such as:

* A verified executable path and publisher
* A specific signed PowerShell script
* A documented automation account
* A defined host group used for dictation or conferencing

I would avoid excluding all PowerShell activity, all microphone ConsentStore events, or entire collaboration applications because those exclusions could create unnecessary blind spots.
